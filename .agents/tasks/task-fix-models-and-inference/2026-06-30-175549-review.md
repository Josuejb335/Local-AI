# Dynamic Hugging Face model discovery and mock fallback removal

This change replaces a hardcoded three-model list with live Hugging Face API discovery (filtering for GGUF files under 2 GB), removes the mock inference fallback from the llama.cpp engine so it fails clearly when native libraries are absent, and deletes broken symlinks from the mis-cased `JniLibs/` directory. The download flow gains size-based validation to prevent false-complete states, and HTTP error responses now carry descriptive messages. The fallback model list (used when the API is unreachable) is updated with known-good community quantizations.

**Watch for:** The HF API's `/models/{id}` siblings array may not include a `size` field for every file (confirmed) — models whose GGUF files lack size metadata will be silently skipped, potentially yielding a shorter list than expected. The sequential 1+N HTTP requests to fetch model details (one per model in the top-50 list) will block the UI-facing flow for several seconds on slow connections (confirmed). The prompt template selection heuristic only recognizes "llama" in the file path (confirmed) — models like TinyLlama that use ChatML despite the name will get the wrong template and produce garbled output.

## High-level view

The repository now issues up to 51 HTTP requests at model-fetch time (1 search + up to 50 detail fetches) sequentially inside `fetchModelsFromHuggingFace()`, so list-loading time scales linearly with network latency. The fallback list kicks in on any exception, which handles network failures gracefully but also masks partial failures mid-iteration — a JSON parse error on model 30 of 50 loses the 29 already-parsed models.

The download validation compares actual file size against the expected `fileSizeBytes` with a 90% threshold. When `fileSizeBytes` is zero (HF didn't report size), the fallback accepts any file over 1 MB — a 1 MB threshold could accept a truncated download of a 1.5 GB model.

The inference engine no longer has any fallback path. If native libraries fail to load, `initialize()` returns `Result.failure`. The prompt template heuristic (`loadedLocalPath.contains("llama")`) is filename-based and will misfire on models that have "llama" in their name but use ChatML format.

## N+1 API request pattern in model discovery

`fetchModelsFromHuggingFace()` hits the search endpoint with `limit=50`, then for each result calls `fetchModelDetail(modelId)` — a synchronous HTTP request to `/api/models/{id}`. On a device with 200ms round-trip to HF, this is 10+ seconds of blocking network before the flow emits anything.

The Hugging Face search API supports `?expand[]=siblings` which inlines the siblings data in the search response. Using that would eliminate the N detail calls entirely. Alternatively, the detail calls could be parallelized with a bounded dispatcher, though that introduces rate-limit risk on HF's side.

If any single model in the loop throws an unexpected exception (e.g., malformed JSON in a sibling entry), the entire function fails and falls back to the hardcoded list. `fetchModelDetail` already wraps its call in try/catch and returns null on failure, but the loop body outside that call (e.g., `findBestGgufFile` parsing) is unprotected. Wrapping the per-model processing in a try/catch would preserve partial results.

## File size gating and the "siblings" size field

```kotlin
val size = sibling.optLong("size", 0)
if (size > 0 && size <= MAX_FILE_SIZE_BYTES) {
    ggufFiles.add(filename to size)
}
```

The HF API does not guarantee a `size` field on every sibling entry. Repositories that haven't had their file metadata indexed will return siblings with `rfilename` but no `size`. These models are silently skipped — `ggufFiles` remains empty, `findBestGgufFile` returns null, and the model is excluded from the list without logging.

This means the dynamic list could be shorter than the user expects ("all compatible models") despite many valid GGUF repos existing on HF. A HEAD request to the resolve URL would provide Content-Length as a fallback, at the cost of additional latency.

On the download side, if `modelInfo.fileSizeBytes` is 0 (siblings didn't report size), the `expectedSize > 0` branch is skipped and the fallback accepts any file over 1 MB. A valid but truncated download of a multi-GB model would pass this check and be marked complete.

## Prompt template selection

```kotlin
val template = if (loadedLocalPath.contains("llama", ignoreCase = true))
    PromptTemplate.LLAMA_3 else PromptTemplate.CHATML
```

This will misfire on TinyLlama (uses ChatML despite the name) and on any HF repo that puts "llama" in the path for unrelated reasons. The GGUF format includes a `tokenizer.chat_template` metadata key that could be read at load time via the native bridge to determine the correct template. The current heuristic works for the fallback list models but becomes increasingly unreliable as the dynamic discovery surfaces a wider variety of model names.

## Download validation edge case

When `modelInfo.fileSizeBytes` is 0 (HF didn't report size at discovery time), both the pre-download existing-file check and the post-download `.tmp` validation fall through to lenient paths: existing files over 1 MB are accepted as complete, and no post-download size check runs. A connection drop mid-download that produces a 2 MB partial file of a 1.5 GB model would be accepted as complete.
