# Real HTTP model download from Hugging Face (v2)

The mock download in `AIModelRepositoryImpl` is replaced with a streaming OkHttp download from Hugging Face. A singleton `OkHttpClient` with generous timeouts is provided via Hilt. Temp-file-then-rename guards against partial downloads. The three blocking issues from v1 — response body leak, missing cancellation, and progress flood — are all fixed.

**Watch for:** `CancellationException` from `ensureActive()` is caught by the generic `catch (e: Exception)` block, which deletes the tmp file and emits `Failed` instead of letting cancellation propagate naturally (confirmed). The `invokeOnCompletion` registration leaks a reference to `call` for the lifetime of the parent job if the download completes normally (likely — minor). Path traversal in `ggufFileName` remains a latent concern once the model list becomes dynamic (possible).

## High-level view

A new concern emerged from the v1 fixes: `ensureActive()` throws `CancellationException`, which is a subclass of `Exception` in Kotlin. The generic `catch (e: Exception)` block catches it, deletes the tmp file, and emits `DownloadState.Failed` — meaning downstream collectors see a "failure" state rather than a clean cancellation. In a `flow {}` builder, `CancellationException` should propagate uncaught so that `flowOn` and the collecting scope handle cancellation correctly. The fix is to add a `catch (e: CancellationException)` that rethrows after cleanup.

The `invokeOnCompletion` handler registered on the parent job holds a reference to the completed `Call` for the lifetime of the ViewModel scope. Harmless for a single download but accumulates dangling handlers if `downloadModel` is called repeatedly. Disposing the handle after `response.use {}` completes would be cleaner.

## CancellationException caught as failure

The read loop calls `coroutineContext.ensureActive()` on each buffer iteration. However, the surrounding try/catch structure defeats it:

```kotlin
try {
    // ... ensureActive() inside loop ...
} catch (e: IOException) {
    tmpFile.delete()
    emit(DownloadState.Failed("Download failed: ${e.message ?: "Unknown IO error"}"))
} catch (e: Exception) {
    tmpFile.delete()
    emit(DownloadState.Failed("Download failed: ${e.message ?: "Unknown error"}"))
}
```

When the coroutine is cancelled mid-download, `ensureActive()` throws `CancellationException`. This is caught by `catch (e: Exception)`, which deletes the tmp file but then emits `DownloadState.Failed` (the download wasn't a failure, it was cancelled) and swallows the exception. Swallowing `CancellationException` breaks structured concurrency: the parent job won't know the child was cancelled, and `flowOn`'s internal machinery may misbehave.

The fix:

```kotlin
} catch (e: CancellationException) {
    tmpFile.delete()
    throw e
} catch (e: Exception) {
    tmpFile.delete()
    emit(DownloadState.Failed("Download failed: ${e.message ?: "Unknown error"}"))
}
```

This preserves tmp-file cleanup while letting cancellation propagate correctly.

## invokeOnCompletion handler lifetime

`coroutineContext.job.invokeOnCompletion { call.cancel() }` registers a completion handler on the parent job. After the download finishes and the flow completes, the handler remains registered until the parent job itself completes. The handler closure holds a reference to the OkHttp `Call` object. If `downloadModel` is called repeatedly within the same ViewModel lifecycle, each completed Call accumulates a dangling handler.

The `invokeOnCompletion` return value is a `DisposableHandle`. Storing it and calling `dispose()` after `response.use {}` completes releases the handler immediately:

```kotlin
val cancelHandle = coroutineContext.job.invokeOnCompletion { call.cancel() }
try {
    val response = call.execute()
    response.use { /* ... */ }
} finally {
    cancelHandle.dispose()
}
```
