# Real HTTP model download from Hugging Face

The mock download loop (delay-based progress simulation) in `AIModelRepositoryImpl` is replaced with a streaming OkHttp download that fetches GGUF model files from Hugging Face. A singleton `OkHttpClient` is provided via a new Hilt module with generous timeouts for large-file transfers. The manifest gains INTERNET and ACCESS_NETWORK_STATE permissions. Temp-file-then-rename protects against partial downloads being mistaken for complete files.

**Watch for:** No cancellation support means a 1 GB download runs to completion even after the user navigates away or the coroutine scope is cancelled (confirmed). Progress floods the UI with an emit per 8 KB buffer read on files approaching 1 GB (confirmed). The response body is never explicitly closed on the non-success path, leaking the connection (confirmed).

## High-level view

The download implementation is straightforward streaming IO: build a URL from model metadata, execute a synchronous OkHttp call on `Dispatchers.IO`, pipe the response body through an 8 KB buffer to a `.tmp` file, rename on success, delete on failure. The approach works for a first pass but omits cooperative cancellation, which means the coroutine's job can be cancelled (e.g., ViewModel cleared) while the blocking `execute()` call and the read loop continue running, consuming bandwidth and battery until the download finishes or the connection times out.

The progress reporting mechanism emits on every 8 KB buffer read. For a 1 GB file, that's ~125,000 emissions. `StateFlow` conflates by default, but throttling to 1% increments would cut emissions from ~125K to ~100 with no visible difference to the user.

The download URL is constructed from `modelInfo.id` and `modelInfo.ggufFileName` via string interpolation. Today these are hardcoded, but once `fetchRemoteModels` hits a real API, a crafted `ggufFileName` containing `../` could write outside the models directory.

<summary>Issues (5)</summary>

1. **No cooperative cancellation** — `okHttpClient.newCall(request).execute()` is a blocking call that ignores coroutine cancellation. Wrap in `suspendCancellableCoroutine` or use OkHttp's `Call.cancel()` tied to the coroutine's job so that navigating away actually stops the download.
2. **Response body leak on HTTP error** — When `response.isSuccessful` is false, `response.body` is never closed. Call `response.close()` (or use `response.use {}`) before returning from the error branch.
3. **Progress emission flood** — Emitting progress on every 8 KB read means ~125K emissions for a 1 GB file. Throttle to percentage-point increments or a time-based window to reduce downstream StateFlow churn.
4. **No resume support for interrupted downloads** — If the download is interrupted (network drop, app kill), the `.tmp` file is deleted and the next attempt restarts from zero. Supporting HTTP Range headers with the existing `.tmp` file would avoid re-downloading hundreds of megabytes.
5. **fetchRemoteModels lost its loading delay without replacement** — Removing `delay(800)` is fine, but `fetchRemoteModels` still emits a hardcoded list synchronously on `Dispatchers.Main`. This works today but will need `flowOn(Dispatchers.IO)` once it makes real network calls to Hugging Face's API.

## Cancellation gap in blocking OkHttp call

The `downloadModel` flow uses `okHttpClient.newCall(request).execute()` which blocks the IO dispatcher thread until the response completes or the read timeout expires. When the ViewModel's `viewModelScope` is cancelled (Activity destroyed, user navigates away), the coroutine hosting this flow is cancelled, but the blocking read loop doesn't check `coroutineContext.isActive` and OkHttp's `Call` is never cancelled.

The practical consequence: on a 1 GB download over a slow connection, the download continues in the background consuming bandwidth and battery for the full 5-minute read timeout. The fix is either:

```kotlin
val call = okHttpClient.newCall(request)
coroutineContext.job.invokeOnCompletion { call.cancel() }
val response = call.execute()
```

Or check `ensureActive()` inside the read loop so the coroutine exits promptly when cancelled.

## Response body leak on error paths

When `response.isSuccessful` is false, the function emits `Failed` and returns without closing the response. OkHttp's response body holds a connection from the pool; not closing it prevents connection reuse and can exhaust the pool under repeated failures (e.g., 404s from a wrong model ID). The idiomatic fix is `response.use { ... }` wrapping the entire response handling, which closes the body on all exit paths.

## Progress emission volume

The inner loop emits `DownloadState.InProgress` on every iteration of the 8 KB buffer read:

```kotlin
while (inputStream.read(buffer).also { read = it } != -1) {
    outputStream.write(buffer, 0, read)
    bytesRead += read
    if (contentLength > 0) {
        val progress = bytesRead.toFloat() / contentLength.toFloat()
        emit(DownloadState.InProgress(progress.coerceIn(0f, 1f)))
    }
}
```

For a 987 MB file (the smallest listed model), that's ~120,000 emissions. A simple guard — only emit when `(progress * 100).toInt()` changes from the previous emission — reduces this to 100 emissions with no behavioral difference to the user.

## Path traversal in filename construction

The download URL and file path are constructed by interpolating `modelInfo.ggufFileName` directly:

```kotlin
val url = "https://huggingface.co/${modelInfo.id}/resolve/main/${modelInfo.ggufFileName}"
val targetFile = File(modelsDir, modelInfo.ggufFileName)
```

Today these come from a hardcoded list, so there's no injection vector. Once `fetchRemoteModels` fetches from an API, a crafted `ggufFileName` like `../../databases/app.db` would resolve outside `modelsDir`. Sanitize the filename (reject or strip path separators) before the API integration lands. This is a latent concern (possible) — not exploitable in the current hardcoded state.
